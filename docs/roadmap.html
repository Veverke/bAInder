Feature Candidates — Appendix C

Cross-referenced against the competitive landscape, existing codebase, and user workflow gaps (March 2, 2026). Not yet assigned to a stage — candidates for v1.x and beyond.

C.1 "Continue in AI" Button in Reader
✕ Abandoned ⏱ Low effort Moderate diff.
A button in the reader header that opens the original chat URL so users can resume the conversation. chat.url is already captured and serialised to frontmatter — the reader simply does not surface it.
⚠️ Previously attempted and abandoned. Copilot (primary dev/test platform) does not expose a stable conversation permalink — the saved URL redirects to a blank new chat or produces inconsistent routing. Revisit only if a platform with stable deep-links (e.g. chatgpt.com/c/<uuid>) becomes the primary target.
C.2 Obsidian Vault Export Format
◌ Candidate ⏱ Medium effort High diff.
A dedicated export mode making the ZIP output directly importable into Obsidian as a vault: [[wikilink]] back-references, YAML tags: mirroring bAInder topic names (activates Obsidian's graph view), and an _index.md per topic folder. No competitor offers this. New format variant in export-engine.js — no structural changes.
C.3 Search Filters
◌ Candidate ⏱ Low–Medium High diff.
Filter pills above the search field: Source (ChatGPT / Claude / Gemini / Copilot), Date range (saved between X and Y), Topic scope (search within a subtree). All operate on data already in storage — no new indexing needed. Note: Stage 8 already lists these; this confirms they're the right priority for that stage.
C.4 Grok + DeepSeek Extractor Support
◌ Candidate ⏱ Medium (per platform) High diff.
ConvoSnap supports Grok and DeepSeek; bAInder does not mention them in the extractor or manifests. Both are surging in adoption and have relatively simple DOM structures. Requires a new case in chat-extractor.js and new content_scripts.matches entries — no architectural changes.
C.5 Passive / Background Auto-Save
◌ Candidate ⏱ High effort Very High diff.
Stage 11 mentions this but provides no design. A MutationObserver in the content script detects when the assistant response settles, checks a configurable word-count threshold, runs duplicate detection, then writes silently with a non-intrusive "Chat auto-saved to [Inbox]. Undo." toast. User opt-in toggle in Settings (off by default). Chat Memo (9 K users, 4.7 ★) is the only competitor doing this today.
C.6 Onboarding / First-Run Walkthrough
◌ Candidate ⏱ Low effort Critical — retention
No first-run experience exists anywhere in the docs or codebase. New users see a blank side panel. A three-step spotlight overlay (single onboarding.js + CSS mask, no library): go to an AI chat page → right-click → Save to bAInder → it appears here, use search to find it. Triggered once when storage has no tree data; onboardingComplete: true suppresses on future opens. Highest ROI item for Chrome Web Store ratings.
C.7 In-Reader Per-Message Copy Button
◌ Candidate ⏱ Low effort Moderate diff.
Each message turn in the reader has no copy affordance. A ⎘ icon injected per message block during renderChat() in reader.js, revealed on :hover, calls navigator.clipboard.writeText() with the plain-text of that turn. ~20 lines of JS + minimal CSS. High daily-use quality-of-life improvement.
C.8 Chat Cross-References / Backlinks
◌ Candidate ⏱ High effort High diff.
Extends the existing annotations.js module to allow a highlight note to reference another saved chat via [[topic/chat title]] syntax, resolved against the live tree on save. A "Related chats" section at the bottom of the reader lists all backlinks to the current chat. Creates a lightweight Zettelkasten / wiki layer. Pairs naturally with the Obsidian export (C.2). No competitor has this.
C.9 Topic Sort Order Control
◌ Candidate ⏱ Low–Medium Moderate diff.
No mechanism for controlling topic order beyond "pinned first." A sort selector in the tree header (persisted to storage): A→Z, date of last activity, chat count, manual drag order. First three modes are a sort pass before tree-renderer.js renders — trivial. Manual drag order is medium effort. Becomes important at 20+ topics.
C.10 Scheduled Backup Reminder
◌ Candidate ⏱ Low effort Trust signal
chrome.storage.local is wiped on profile reset or reinstall. After 30 days without a ZIP export, show a dismissible banner: "47 saved chats · Last exported 32 days ago · Export now." One click triggers the existing ZIP export. Tracks lastExportTimestamp in storage. Addresses the known data-loss risk of local-only storage and signals production quality to new users.
C.11 Multi-Platform AI Site Validation
◌ Candidate ⏱ Medium (per platform) High diff.
The extractor declares support for ChatGPT, Claude, Gemini, and Copilot, but end-to-end testing and day-to-day development have been conducted almost exclusively on Copilot. Each platform has a distinct DOM structure that evolves over time; extractors for the other sites are likely broken or partially degraded.

Work items:
  • Manual smoke-test each platform against the current extractor (chat-extractor.js).
  • Write platform-specific fixtures and regression tests (extend chat-extractor.test.js).
  • Fix any selector drift discovered for ChatGPT, Claude, Gemini.
  • Add CI artefacts (HTML snapshots) so future DOM changes are caught automatically.
  • Consider adding Perplexity, Grok, and DeepSeek (see C.4) in the same pass.
⚠️ Known gap. Until this is resolved, bAInder should be described as "Copilot-tested" rather than "multi-platform" in store listings and marketing copy.
C.12 Firefox & Edge Browser Support
◌ Candidate ⏱ Medium–High effort High diff.
bAInder is built on webextension-polyfill which in principle supports Chrome, Edge, and Firefox, but only Chrome has been meaningfully tested. Shipping on Firefox and Edge unlocks a substantial user base with no additional AI-site work.

Work items:
  • Edge: Chromium-based — likely works today; run smoke tests and fix any Manifest V3 or side-panel API gaps. Publish to the Microsoft Add-ons Store.
  • Firefox: Requires a separate MV2 (or MV3-beta) manifest variant; see docs/edge-porting-plan.md for the porting strategy already drafted. Key risks: chrome.sidePanel has no Firefox equivalent — a popup or browser_action fallback is needed; service-worker vs. background-page difference.
  • Set up separate CI matrix jobs for Chrome, Edge, and Firefox (web-ext lint + unit tests).
  • Maintain separate manifests under src/manifests/ per browser target.
ℹ️ The docs/edge-porting-plan.md document describes the Edge path in detail. Firefox adds meaningful scope; estimate 4–6 additional weeks on top of the Edge work.
